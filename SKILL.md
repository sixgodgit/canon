# Canon — 技能生态典章 v2.1

> **图书管理员不烧书，园丁不灭种，审计员不执刑。**
>
> Canon 的核心不再是判断 skill 是否该活，
> 而是判断它应该处在哪个生态位：舞台、书架、冷库、种子库、禁区。
>
> 真正的智能不是更会删除，而是更会安置。

**独立项目仓库：** `sixgodgit/canon` — 完整技能目录（含 references/、分类 JSON、决策树、金句集）。更新推到此仓库，不是冷存储。

## 核心定位

v1.0 是**园丁**（Gardener）— 用剪刀修剪花园，决定谁活谁死。
v2.0 是**图书管理员**（Librarian）— 分类上架，决定书放在哪个区。

> **真正的园丁不是决定谁该死，而是决定谁该上台、谁该入库、谁该封存、谁只能在受控实验室里被阅读。**

核心转变：
- 没有技能被"删除"——只有被重新分类
- 所有低分技能进入种子库（可复活）而非坟墓
- 危险技能进入隔离层（可借阅）而非处决
- 最终用户（你和 GPT）自己从不同分区借书

## 四层架构

```
┌─────────────────────────────────────────────────┐
│              🚀 热气层（常备能力）                │
│  重启即加载 · 追求效率 · 极端熵减 · 高频调用     │
│  只放最活跃、最可靠的 10-15 个技能                │
│  性能敏感，任何延迟都必须从这层移除                │
├─────────────────────────────────────────────────┤
│              📚 常驻层（知识书库）                  │
│  按需懒加载 · 可冗余 · 参考索引 · 中频调用        │
│  大部分 keep 技能在这里，有索引但不常驻内存         │
│  加载成本 100-500ms，可接受                       │
├─────────────────────────────────────────────────┤
│              🌱 种子库（历史可能）                  │
│  不加载 · 仅存元数据索引 · 零运行时成本             │
│  所有 cold_storage 和 probation 技能在这里         │
│  anti_entropy_flag=true 的技能必入库               │
│  定期（每 30 天）随机抽 1-2 个放回常驻层观察        │
│  禁止永久删除，只有"入库时间"和"最后借阅时间"       │
├─────────────────────────────────────────────────┤
│              🔒 隔离层（危险知识）                  │
│  需显式授权 + 审计日志 · 只读 · 每次调用留记录      │
│  godmode、obliteratus 等 red-teaming 技能          │
│  不销毁知识，但要求每一次借阅都有理由和记录          │
│  红队测试时可用，日常环境中不可见                   │
│  完整访问控制/审计日志格式/沙箱建议见               │
│  references/isolation-policy.md                    │
└─────────────────────────────────────────────────┘
```

隔离层的详细安全策略（访问控制、审计日志字段、沙箱执行配置示例）不在本文档展开，统一维护于 [references/isolation-policy.md](references/isolation-policy.md)，避免规则散落导致更新遗漏。

## 执行流程

### 步骤 0：数据采集 — 三源核对

每周健康检查/手动评估时，必须从三个来源收集数据以避免遗漏：

1. **skills_list()** — Hermes 可见的活跃技能
2. **磁盘扫描** — `find ~/.hermes/skills -name "SKILL.md"`
3. **旧分类文件** — `references/librarian-classification.json`

### 步骤 1：加载输入

读取 skill-ecosystem-assessment.json（v1.0 的输出或手动评估结果）。
如果是每周 cron job，读取 `references/librarian-classification.json` 作为基准。

### 步骤 2：四层分类

对每个技能不做"生死判决"，而是分配馆藏层级：

| 原 recommended_action | 映射到 | 说明 |
|:---|:---|:---|
| keep (score >= 80) | 🚀 热气层 | 核心能力，高频使用 |
| keep (score 50-79) | 📚 常驻层 | 正常技能，按需加载 |
| probation | 🌱 种子库 | 有潜力待观察 |
| cold_storage | 🌱 种子库 | 低活跃但保留可能性 |
| anti_entropy_flag=true | 🌱 种子库（强制） | 多样性保护，禁止升层 |
| delete_candidate (危险) | 🔒 隔离层 | 危险但保留知识 |
| delete_candidate (重复) | 🌱 种子库 + 合并标记 | 重复但保留为历史记录 |

### 步骤 2.5：热度检测与自动轮转

在四层分类之后、随机轮转之前，先运行 `engine/` 下的自动化引擎，用真实使用数据校正人工分类结果：

1. **运行热度分析**：`python3 engine/heatmap.py`
   - 读取 `~/.hermes/skills/.usage.json`
   - 对每个技能计算总调用次数、近7天/近30天活跃度、趋势（hot/active/cooling/cold/stale）
   - 输出 Top 20 热门技能与 Bottom 10 冷门技能，并导出 `heatmap-report.json`

2. **运行晋升/降级引擎**：`python3 engine/promotion.py`
   - 高频规则：近 30 天调用 > 50 次 → 建议升级至 🚀 热气层
   - 低频规则：近 90 天调用 < 3 次 → 建议降级入 🌱 种子库
   - `anti_entropy_flag=true` 的技能不参与自动降级（继承 decision-tree.md 的多样性保护规则）
   - 🔒 隔离层技能永不参与自动晋升/降级，规则见 `references/isolation-policy.md`
   - 热气层设有容量守卫（默认 15 个），晋升建议超出容量时按近 30 天调用量排序，仅保留前 N 个高置信度建议，其余进入观察队列
   - 引擎只产出建议（`promotion-recommendations.json`），**不直接改写** `librarian-classification.json`，最终层级变更需人工确认或由 Canon 主流程二次校验后写入

3. **将建议并入本轮评估**：把 `promotion.py` 的输出与步骤 2 的人工/历史分类结果合并，作为步骤 3 随机轮转的输入基准，而不是凭直觉判断谁该升谁该降。

> 数据说明：当前 `.usage.json` 只记录累计 `use_count` 和 `last_used_at` 等快照字段，没有逐次调用的时间序列，因此近 30/90 天调用数是**估算值**（详见 `engine/promotion.py` 中 `estimate_recent_calls` 的注释）。如果未来 `.usage.json` 升级为记录时间序列日志，应替换为精确统计。

### 步骤 3：热度感知轮转（反熵减机制）

轮转不再是纯随机抽取，而是结合步骤 2.5 的热度数据做出更有依据的调整：
- 从种子库中，优先选择 `heatmap.py` 判定为回暖趋势（如近期被重新查看/借阅）的技能放回常驻层观察，而非完全随机抽取
- 若步骤 2.5 未发现回暖信号，仍按原规则从种子库随机抽取 1-2 个技能放回常驻层观察，保留反熵减的兜底机制
- 从隔离层检查是否有技能可降级——但隔离层的任何层级变更必须遵循 `references/isolation-policy.md` 中的人工审批流程，不因热度数据自动执行

### 四层行为对照表

| 行为 | 🚀 热气层 | 📚 常驻层 | 🌱 种子库 | 🔒 隔离层 |
|:---|:---|:---|:---|:---|
| 启动时加载 | ✅ 是 | ❌ 否 | ❌ 否 | ❌ 否 |
| 按需加载 | ✅ 是 | ✅ 是（懒加载） | ❌ 否 | ❌ 需授权 |
| 可被 Hermes 自动发现 | ✅ 是 | ✅ 是 | ❌ 索引可见 | ❌ 不可见 |
| 性能影响 | 直接影响启动速度 | 不影响启动 | 零影响 | 零影响 |
| 可被删除 | ❌ 不可 | ❌ 不可 | ❌ 不可 | ❌ 不可 |
| 可被借阅 | 随时 | 随时 | 需正常加载 | 需授权+审计 |
| 最大容量 | ~15 个 | 无限制 | 无限制 | 无限制 |

- **v2.0**: 从 Gardener 升级为 Librarian。四层架构。品牌名从 skill-ecosystem-librarian 更改为 Canon。
- **v1.0** (已归档): Gardener 版本。以"园丁"角色做判断。
